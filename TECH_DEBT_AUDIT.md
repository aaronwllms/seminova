# Tech Debt Audit — Seminova

Generated: 2026-06-23

## Executive summary

- **Starter scaffold still ships in production paths** — `useGetMessage`, `axios`, MSW `/api/message` handler, and `auth-button.tsx` are dead demo code with no route or consumer; remove before the next product spinoff inherits it.
- **Duplicate Radix dependency graph** — UI imports the `radix-ui` umbrella package while `package.json` also lists four `@radix-ui/react-*` packages that nothing imports; knip/depcheck both flag them.
- **God files violate the ≤150-line locked rule** — `sidebar.tsx` (726 LOC), `profile-settings-form.tsx` (320), and `users/actions.ts` (299) are the worst offenders; churn will concentrate here.
- **Coverage gates mask admin shell and service client** — `vitest.config.ts` excludes admin layout chrome and `service.ts` sits at 0% in-scope coverage while admin promote/demote depends on it.
- **`ReactQueryDevtools` ships in the root layout unconditionally** — adds client bundle weight to every page including marketing.
- **No `error.tsx` route boundaries** — fault paths rely entirely on inline `ErrorPanel`; a thrown server error in a layout has no dedicated recovery UI.
- **Auth API inconsistency (`getClaims` vs `getUser`)** — admin paths use JWT claims; profile/avatar paths use session user; workable but undocumented at call sites and easy to get wrong when adding routes.
- **Epic plans still cite `src/app/(admin)/`** — Phase 6 moved to `src/app/admin/` but `.cursor/plans/` and `CONTEXT_ARCHIVE.md` still reference the old invisible route group.
- **Quality gates pass** — `pnpm audit` clean; `type-check`, `lint`, `test:ci` all green (152 tests, ~87% coverage).
- **No declared `// debt:` markers** — `code-minimalism.mdc` convention not yet adopted in `src/`; nothing to harvest under Declared debt.

## Architectural mental model

Seminova is a **Next.js 16 App Router template** (not a finished product) organized into route groups: public `(marketing)/` at `/`, `auth/` at `/auth/**`, authenticated `(app)/` at `/profile`, and `admin/` at `/admin/**`. Session refresh and the auth boundary run in `proxy.ts` → `src/supabase/proxy.ts`; admin role gating is duplicated in the proxy (redirect) and `AdminAuthGate` (layout). Data access splits three ways: browser client (`@/supabase/client` + RLS), server session client (`@/supabase/server`), and secret-key service client (`@/supabase/service`) for admin user listing and role mutations. UI is shadcn-owned primitives + shared chrome (`site-*`) + route-scoped `_components`. Config-driven identity lives in `src/config/site.ts` and `landing-content.ts`. The repo is young (~60 commits since initial scaffold) with heavy doc/rules investment and Phase 1–6 feature delivery complete; git churn concentrates on docs (`CONTEXT.md`, `AGENTS.md`) and auth/admin/profile surfaces.

**Hot paths:** `proxy.ts`, auth forms, `getCurrentUserProfile`, profile blur-save, admin users table + server actions, avatar upload pipeline.

**Cold corners:** TanStack Query demo hook, `auth-button`, unused shadcn primitives (`checkbox`, `collapsible`), `ThemeProvider` wrapper, MSW browser entrypoints.

**Largest files (LOC):** `sidebar.tsx` (726), `profile-settings-form.tsx` (320), `users/actions.ts` (299), `dropdown-menu.tsx` (257), `users-table.tsx` (231).

**Most modified (6 months):** `CONTEXT.md`, `AGENTS.md`, `README.md`, auth forms, `layout.tsx`, `vitest.config.ts`.

## Things that look bad but are actually fine

- **Dual admin gating (`proxy.ts` + `AdminAuthGate`)** — Looks redundant, but Phase 6 explicitly chose defense-in-depth: proxy rejects early at the edge; layout gate catches test/dev bypass. Keep both unless proxy becomes the sole enforcement layer by design.

- **`getClaims()` in proxy/admin vs `getUser()` in profile** — Supabase documents `getClaims()` for middleware/session refresh performance; `getUser()` validates against Auth server for sensitive mutations (avatar upload, profile update). The split is sound — it just needs documenting (F021), not unification.

- **Auth forms on `useState` while profile uses RHF+zod** — `forms.mdc` explicitly defers auth migration. Migrating login/sign-up to RHF would be churn without UX benefit.

- **`data-table1.tsx` filename** — Ugly, but AGENTS.md and `data-tables.mdc` canonize it as the reference implementation; renaming is cosmetic (F009), not structural debt.

- **Coverage exclusions for `page.tsx` / `layout.tsx` shells** — Thin re-exports with no logic; excluding them from the denominator is reasonable. Excluding `_components` with real logic (admin shell) is not — see F028.

- **Bracket-tagged `console.error` instead of a logger module** — `AGENTS.md` and `logging.mdc` lock this pattern for Vercel searchability; not debt.

- **`APP_HOME === PROFILE_PATH` with comment about future divergence** — `app-paths.ts:3-9` documents the intentional collapse; acceptable until a separate app home exists.

- **Thin marketing wrappers (`LandingFooter`, etc.)** — One-liner components look pointless but establish a consistent import boundary for products that may fork marketing copy independently of shared chrome.

- **No `src/services/` repository layer yet** — `supabase.mdc` recommends it for future Supabase queries; only two tables exist. Premature abstraction would violate code-minimalism.

- **`pnpm audit` / CI green with act() stderr noise** — Tests pass; warnings are hygiene (F032–F034), not blocking failures.
